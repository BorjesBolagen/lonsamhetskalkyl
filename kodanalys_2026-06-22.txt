--------------------------------------------------------------------------------
HÖG PRIORITET
--------------------------------------------------------------------------------

[HÖG] H1. Saknad auktorisering på /api/users/get/user (informationsläckage)
  Fil: web/app/api/users/get/user/route.ts
  GET-routen tar emot ett userId som queryparameter och returnerar HELA
  User-raden (select("*")) utan att kontrollera vem som anropar. Klienten skapas
  med getSupabaseServerClient() (anon-nyckel + cookies), så skyddet beror helt
  på att Row Level Security (RLS) i Supabase är korrekt konfigurerad. Om RLS
  saknas eller är för tillåtande kan vilken inloggad (eller oinloggad) användare
  som helst läsa andra användares data genom att gissa/iterera userId.
  Åtgärd: Verifiera anropande användare (admin eller samma id) innan data
  returneras, och bekräfta att RLS-policy finns för User-tabellen. Begränsa även
  vilka kolumner som returneras (undvik select("*")).

[HÖG] H2. Säkerhet i lönsamhetskedjan vilar enbart på odokumenterad RLS
  Fil: web/app/api/profitability/route.ts
  Den omskrivna routen gör flera databasuppslag (tax_point_lookup) och kör
  affärskritisk prisberäkning helt utan explicit autentiseringskontroll. Som
  ovan beror skyddet på RLS. Affärslogik och prissättningsdata bör inte vara
  åtkomlig för oinloggade.
  Åtgärd: Lägg till en central auth-helper (se M5) och anropa den i början av
  routen. Dokumentera tydligt vilka tabeller som skyddas av RLS.

[HÖG] H3. Division med noll i prisestimering ger NaN/Infinity i ekonomiska tal
  Fil: web/profitability/trappsteg_steg.ts (funktionen estimeraPris, ~rad 60-65)
    return totalKundnettofrakt / totalVikt * weight;
  Om totalVikt = 0 (alla rader har vikt 0, eller tom datamängd som ändå når hit)
  blir resultatet Infinity eller NaN, vilket sedan kan spridas vidare genom
  intäkts-/lönsamhetsberäkningen och visas för användaren. Se även relaterad
  null-hantering i try_steg_4 där Math.min() kan ta emot null/NaN.
  Åtgärd: if (totalVikt === 0) return null; och säkerställ att efterföljande steg
  hanterar null innan Math.min()/jämförelser.

MEDEL PRIORITET
--------------------------------------------------------------------------------

[MEDEL] M1. Otypade Supabase-anrop med (supabase as any)
  Fil: web/app/api/profitability/route.ts (flera ställen kring tax_point_lookup)
  Castning till any kringgår TypeScript och döljer eventuella schema-/typfel.
  Åtgärd: Lägg in tax_point_lookup i det genererade Supabase-schemat och ta bort
  as any så anropen blir typade.

[MEDEL] M2. N+1 / upp till 4 sekventiella DB-anrop per lönsamhetsberäkning
  Fil: web/app/api/profitability/route.ts
  Routen gör upp till fyra separata await-anrop mot tax_point_lookup
  (destination postnr -> destination postort -> pickup postnr -> pickup postort),
  helt sekventiellt. Det blir långsamt och belastar databasen, särskilt om routen
  anropas en gång per sändning i en lista.
  Åtgärd: Slå ihop till en RPC/batch-fråga, eller hämta alla aktuella taxepunkter
  i ett anrop. Cacha tax_point_lookup (data ändras sällan).

[MEDEL] M3. Validera taxPointRelation-format innan parsning
  Fil: web/profitability/trappsteg_steg.ts (~rad 21)
    const [sender_taxep, receiver_taxep] = input.taxPointRelation?.trim()
        .split("-").map(Number) || [];
  Vid t.ex. "123-456-789" eller tomt värde blir resultatet fel/undefined utan
  felhantering, vilket ger felaktig taxepunktsmatchning.
  Åtgärd: Validera att split ger exakt två delar och att båda är giltiga tal;
  annars hoppa över steget / returnera tydligt fel.

[MEDEL] M4. Bräcklig delning av data mellan steg via _isDbValidDestination
  Filer: web/app/api/profitability/route.ts, web/profitability/service.ts
  Routen lägger till en informell property (consignment as any)._isDbValidDestination
  som determineFlowType()/routeConsignment() sedan läser via (consignment as any).
  Anropas dessa funktioner från annat håll saknas flaggan och flödet kan
  felklassificeras.
  Åtgärd: Gör fältet till en officiell del av typen (eller skicka det som separat
  argument) så kontraktet blir explicit och typkollat.

[MEDEL] M5. Duplicerad auth-/felhanteringskod i många routes
  Filer: web/app/api/users/route.ts, web/app/api/messages/add/route.ts m.fl.
  Samma mönster (getCurrentUser -> 401 -> kontrollera role admin -> 403)
  upprepas i route efter route, med små variationer. Vissa routes saknar det helt
  (H1, H2).
  Åtgärd: Skapa en helper, t.ex. requireUser()/requireAdmin(), och använd den
  konsekvent. Minskar både duplicering och risken att glömma kontrollen.

[MEDEL] M6. Duplicerade utility-funktioner mellan moduler
  Filer: web/app/home/hooks/homeTypesAndUtils.ts och
         web/lib/backend/transportPlanningUtils.ts
  Funktioner som calculateConsignmentProfitabilityPrice, toBarPercent,
  orientLineNameForSelectedAreas och getDisplayCustomerName finns på flera
  ställen med likartad logik. Risk för att de divergerar över tid.
  Åtgärd: Konsolidera till en delad modul.

[MEDEL] M7. Ingen caching av externa iLog-anrop
  Filer: web/app/api/ilog/lines/route.ts, equipages/route.ts, consignments/route.ts
  Externa iLog-anrop görs utan caching; samma linjer/ekipage hämtas upprepade
  gånger per session.
  Åtgärd: Lägg på lämplig Cache-Control / revalidate för stabil data, eller cacha
  i minne/Redis.
