2. SAMMANFATTNING AV KODANALYSEN
--------------------------------------------------------------------------------
Kodbasen (~16 000 rader TS/TSX/JS) är överlag välskriven: inga `: any`-typer
hittades, getCurrentUser-mönstret används konsekvent i känsliga endpoints,
admin-operationer (radera användare, historisk import) har korrekt rollkontroll
på serversidan, och service role-nyckeln används bara på servern via
getSupabaseAdminClient.

Det största problemet är att proxy.ts (middleware) endast skyddar SIDOR
(/home, /settings, /admin m.fl.) – inte /api/*. Flera API-routes saknar därför
helt autentiseringskontroll och förlitar sig enbart på Supabase RLS, vilket
inte hjälper för data som hämtas från externa system (iLog) eller beräknas i
servern (profitability). Därutöver finns ett antal prestanda- och
kvalitetsproblem, främst i de mycket stora simulator-filerna.

3. PRIORITERADE FÖRBÄTTRINGSFÖRSLAG
--------------------------------------------------------------------------------

=== HÖG PRIORITET ===

[H1] SÄKERHET: iLog-endpoints saknar autentisering
     Filer: web/app/api/ilog/{equipages,lines,consignments,consignment,
            unassigned-consignments}/route.ts
     Ingen getCurrentUser-kontroll. Endpoints hämtar affärskänslig
     transportdata från iLog och kan anropas oautentiserat (proxy.ts skyddar
     inte /api/*, och RLS gäller inte data som proxas från iLog).
     Åtgärd: Lägg till auth-kontroll i varje route (eller gemensam wrapper):
       const currentUser = await getCurrentUser(await getSupabaseServerClient());
       if (!currentUser.status) return 401.

[H2] SÄKERHET: GET /api/users och GET /api/users/get/user saknar auth och
     returnerar select("*")
     Filer: web/app/api/users/route.ts:19-36,
            web/app/api/users/get/user/route.ts:13-39
     Ingen kontroll av vem som frågar; vilken klient som helst kan slå upp
     godtycklig användares fulla rad (e-post, roll, filter) via ?userId=.
     RLS kan mildra detta men endpoints bör inte förlita sig på det ensamt.
     Åtgärd: Kräv inloggning; för /get/user kräv admin ELLER att userId ==
     inloggad användare. Begränsa select() till nödvändiga fält.

[H3] SÄKERHET: debugRaw-parameter exponerar rå iLog-data i produktion (PÅ MAIN)
     Filer: alla fem ilog-routes (t.ex. equipages/route.ts:38-44)
     ?debugRaw=true returnerar omappad rådata från iLog utan kontroll.
     Åtgärd: REDAN FIXAT i commit f5572d5 på denna branch – merga till main.

[H4] SÄKERHET: /api/token returnerar hela sessionsobjektet
     Fil: web/app/api/token/route.ts:12
     `token: session` exponerar access_token/refresh_token i JSON-svaret
     trots att tokens redan hanteras via httpOnly-cookies. Kommentaren i
     filen säger att den bara ska svara om token är giltig.
     Åtgärd: Returnera endast { status: true } utan sessionsobjektet.

[H5] SÄKERHET: 25 kända sårbarheter i beroenden (Dependabot)
     GitHub rapporterar 11 höga, 12 medelhöga och 2 låga sårbarheter i
     projektets beroenden på default-branchen.
     Åtgärd: Gå igenom https://github.com/BorjesBolagen/lonsamhetskalkyl/
     security/dependabot och uppdatera paketen (npm audit fix / bumpa
     versioner i web/package.json).

[H6] BUGG: Delade mutabla variabler i simulatorns körning (race condition)
     Fil: web/app/simulator/useSimulatorPlanner.ts:~705-989
